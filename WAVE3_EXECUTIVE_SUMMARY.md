# Wave 3 `any` Type Elimination: Executive Summary

## Overview

This Wave addresses TypeScript `any` type usage across the codebase to improve type safety and catch potential bugs at compile-time rather than runtime.

---

## The Situation

**Current State:**
- 624+ instances of `any` across 65 files
- 184 HIGH-risk instances (database queries, return types)
- 180 MEDIUM-risk instances (function params, type casting)
- 65 LOW-risk instances (catch clauses, simple variables)

**Why It Matters:**
1. Database schema changes cause silent runtime failures
2. Refactoring becomes unsafe (no compiler feedback)
3. IDE autocompletion is ineffective
4. Code review becomes harder (no type contracts)
5. New developers struggle to understand data structures

## Key Actions by Category

### Catch Clauses (25 instances) - EASY
```typescript
// Before
catch (error: any) { }

// After
catch (error: Error) { }
```
**Time: 1 hour | Risk: NONE**

### Database Queries (120 instances) - CRITICAL
```typescript
// Before
const rows = await dbQuery(...) as any[];

// After
const rows: FamilyRow[] = await dbQuery(...);
```
**Time: 6 hours | Risk: HIGH (must get schemas right)**

### Function Parameters (95 instances) - IMPORTANT
```typescript
// Before
export async function createProduct(data: any) { }

// After
export async function createProduct(data: CreateProductPayload) { }
```
**Time: 8 hours | Risk: MEDIUM**

### Type Casting (85 instances) - MODERATE
```typescript
// Before
const p = product as any;

// After
const p: ComposedProduct = product;
```
**Time: 8 hours | Risk: MEDIUM (context-dependent)**

### Array Types (40 instances) - EASY
```typescript
// Before
const rules: any[] = [];

// After
const rules: NamingRule[] = [];
```
**Time: 4 hours | Risk: NONE**

### React Props (35 instances) - IMPORTANT
```typescript
// Before
interface ModalProps {
  product: any;
}

// After
interface PostSaveExportModalProps {
  product: ComposedProduct;
}
```
**Time: 4 hours | Risk: NONE**

## Risk Assessment

### What Could Go Wrong
1. ❌ Prisma-generated types change format
   - **Mitigation:** Don't edit generated code
   
2. ❌ Type errors prevent build
   - **Mitigation:** Incremental changes, test after each phase
   
3. ❌ Missed imports break references
   - **Mitigation:** Run `npm run build` frequently
   
4. ❌ Database schema changed without updating types
   - **Mitigation:** Keep types in sync with schema

### Why This is Low-Risk
✅ Type-only changes (no runtime modifications)
✅ No breaking changes to public APIs
✅ Backward compatible with existing code
✅ Can be reverted incrementally
✅ No database migrations needed
✅ No dependency updates required
